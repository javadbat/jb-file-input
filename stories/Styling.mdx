<Title>Styling</Title>

`JBFileInput` styling is theme-driven. Use common design tokens for product-level consistency, component variables for the file input API, and CSS parts only for exposed text and icon surfaces that need targeted styling.

## Common Design Tokens

Use the core theme docs as the source of truth for shared design-system variables:

- [Theme Colors](../../jb-core/theme/stories/colors.mdx) explains the general color palette variables, default colors, customization approaches, and color generation helpers.
- [Theme Sizes](../../jb-core/theme/stories/sizes.mdx) explains shared size and radius variables.

File input recipes should use those common tokens where possible, then map them into `--jb-file-input-*` variables only where the file input needs a component-specific value.

## Styling Model

Use three layers when styling file input examples:

- Common design tokens are documented in [Theme Colors](../../jb-core/theme/stories/colors.mdx) and [Theme Sizes](../../jb-core/theme/stories/sizes.mdx).
- Reusable example theme packs live in [`docs/styles`](https://github.com/javadbat/design-system/tree/main/docs/styles) and can be shared across components.
- Component recipe files live in [`modules/jb-file-input/stories/styles`](https://github.com/javadbat/design-system/tree/main/modules/jb-file-input/stories/styles).
- Public component variables use the `--jb-file-input-*` prefix and are the stable styling API.

Each example uses one shared example theme pack from [`docs/styles`](https://github.com/javadbat/design-system/tree/main/docs/styles) and one file-input-specific recipe from [`modules/jb-file-input/stories/styles`](https://github.com/javadbat/design-system/tree/main/modules/jb-file-input/stories/styles). The style story applies both classes directly to `JBFileInput`.

## Recommended Customization Path

1. Use `--jb-file-input-*` variables first for the default placeholder, selected-file, uploading, and overlay surfaces.
2. Use host styles for layout context such as width, height, borders, and shadows around the component.
3. Use common theme variables from [Theme Colors](../../jb-core/theme/stories/colors.mdx) and [Theme Sizes](../../jb-core/theme/stories/sizes.mdx) when the style should align with the design system.
4. Put reusable theme-pack values in [`docs/styles`](https://github.com/javadbat/design-system/tree/main/docs/styles) only when they should be shared by multiple component demos.
5. Use `::part(...)` sparingly for official exposed surfaces such as `::part(placeholder-title)`, `::part(uploading-title)`, `::part(file-name)`, `::part(placeholder-icon)`, and `::part(file-icon)`.
6. Add a new component variable only when a needed internal surface cannot be customized with the current public variables or official parts.

## Theme File Pattern

Use [Theme Colors](../../jb-core/theme/stories/colors.mdx) and [Theme Sizes](../../jb-core/theme/stories/sizes.mdx) as the common design-token guides. Theme-pack files should keep reusable demo variables in [`docs/styles`](https://github.com/javadbat/design-system/tree/main/docs/styles), then each component should keep its own style CSS in its story [`styles`](https://github.com/javadbat/design-system/tree/main/modules/jb-file-input/stories/styles) folder.

```css
/* https://github.com/javadbat/design-system/blob/main/docs/styles/carbon.css */
.theme-name-style {
  /* shared design-system variables */
}

/* https://github.com/javadbat/design-system/blob/main/modules/jb-file-input/stories/styles/style-carbon.css */
.theme-name-file-input {
  /* file-input-specific variables and official parts */
}
```

This keeps common theme tokens reusable and lets each component page show only the CSS it needs.
